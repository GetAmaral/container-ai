# Story 1.6: Tratar Reembolso e Chargeback

## Status
Draft

## Executor Assignment
executor: "@dev"
quality_gate: "@architect"

## Story
**As a** sistema backend,
**I want** desativar automaticamente o acesso de users que pediram reembolso ou fizeram chargeback,
**so that** users que nao sao mais clientes percam acesso imediatamente.

## Acceptance Criteria

1. Ao receber webhook PURCHASE_REFUNDED, buscar user por order_id e desativar acesso
2. Ao receber webhook CHARGEBACK, buscar user por order_id e desativar acesso
3. Status do user atualizado para "desativado"
4. Data de desativacao registrada no banco (desativado_at, motivo)
5. Se user nao encontrado para o order_id, logar warning e ignorar (nao falhar)
6. Idempotencia: se user ja esta desativado, nao reprocessar
7. (Opcional) Enviar mensagem informando o user sobre a desativacao

## Dev Notes

### Motivos de Desativacao
```
refund     -> User pediu reembolso e foi aprovado
chargeback -> User contestou cobranca no cartao
```

### Diferenca entre Refund e Chargeback
- **Refund**: User pediu devolucao, processo amigavel
- **Chargeback**: User contestou no banco/cartao, pode indicar fraude

Ambos desativam acesso, mas chargeback deve ser logado com mais detalhe (possivel fraude).

### Notas Importantes
- Webhook de refund/chargeback pode chegar dias ou semanas apos a compra
- User pode ja ter sido ativado e estar usando o produto
- Desativacao deve ser imediata ao receber o evento
- Considerar: o que acontece com dados do user? (manter no banco, so mudar status)

### Testing
- Testar cenario temporal: user ativado ha dias, chega refund -> desativa
- Testar webhook duplicado (idempotencia)
